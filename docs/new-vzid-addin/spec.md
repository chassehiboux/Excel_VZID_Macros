# Спецификация: новая надстройка VZID

## Статус

Черновик для согласования с бизнес-заказчиком.

## Цель

Собрать новую надстройку Excel, которая:

- запускается предсказуемо и без самописного "опасного" автообновлятора;
- обновляется через GitHub Releases;
- даёт единый интерфейс для всех регионов;
- уменьшает количество отдельных стартовых окон и разрозненных сценариев;
- упрощает поддержку, выпуск обновлений и диагностику проблем.

## Как это должно выглядеть для пользователя

Пользователь открывает Excel и видит один понятный вход в систему макросов.

Вместо отдельных стартовых форм по регионам пользователь работает через единый интерфейс:

- выбирает регион работы;
- видит только подходящие для этого региона действия;
- запускает нужный макрос одной кнопкой;
- при необходимости открывает отдельное окно только для тех сценариев, где без окна неудобно.

## Согласованные решения

- Делать новый проект надстройки, а не продолжать развивать текущую архитектуру.
- Уйти от стартовой схемы с отдельными региональными `UserForm` как главной точкой входа.
- Сделать единый интерфейс выбора региона и команд.
- Обновление строить через GitHub Releases.
- Допускать отдельные `UserForm` только там, где действительно нужно вводить параметры или показывать сложный сценарий.
- Обновление для пользователя делать через мягкое уведомление: если доступна новая версия, показать уведомление и кнопку обновления.
- Базовый вариант источника обновлений: прямой доступ к GitHub.
- Основной интерфейс делать как отдельную вкладку `VZID` на ленте Excel.
- Установка должна работать без прав администратора.
- Поставка продукта должна идти как один общий пакет для всех регионов.
- Первый способ установки должен быть через один `setup.exe`, который пользователь запускает без прав администратора.
- Выбранный регион должен запоминаться между запусками.
- Должна быть возможность ограничивать доступ к отдельным кнопкам по Windows-логину.
- Пользователь `dzuikevich` должен иметь максимальные права по умолчанию.
- Обновление должно применяться только после полного закрытия всех окон Excel и нового запуска.
- После скачивания обновления пользователю нужно предлагать перезапуск Excel прямо сейчас с понятным предупреждением о том, когда именно обновление вступит в силу.
- Код, установщик, релизы и файлы обновления хранятся в одном публичном репозитории.
- Если GitHub недоступен, пользователь продолжает работу без блокировки, а на вкладке `VZID` показывается маленький alert о том, что проверить обновление не удалось.
- Нужно предусмотреть два способа обновления:
  через GitHub Releases как основной путь и через локальный файл как резервный путь.
- Команды без доступа должны быть видимы, но неактивны.
- Правила доступа на старте хранятся прямо в конфиге надстройки.
- Конфиг надстройки должен храниться в `%AppData%\Microsoft\Excel\LocalCache\VZID\config\config.json`.
- В качестве базовой схемы принимаем одну рабочую Excel-надстройку `MainVZID.xlam` без отдельного постоянного `Loader`.
- `MainVZID.xlam` должен подключаться к Excel как обычная пользовательская надстройка.
- Обновление должен выполнять отдельный `updater.exe`, который ставится вместе с `setup.exe`, запускается только по явному сценарию обновления и может ждать полного закрытия Excel в фоне.
- На вкладке `VZID` в MVP должны использоваться общие по смыслу кнопки, а доступность и действия должны определяться выбранным регионом.
- В dropdown региона должны быть `Курган`, `Тюмень`, `Екатеринбург`, `ЧЭС`.
- На вкладке `VZID` должны быть отдельные блоки `Формирование сопроводительных документов`, `Действия`, `Обновление`.
- Кнопки, для которых макрос не задан в конкретном регионе, должны оставаться видимыми, но неактивными.
- Две специальные кнопки Тюмени должны показываться только при выборе региона `Тюмень`.
- В блоке обновления должен отображаться реальный номер установленной или подготовленной версии, а не текст `версия актуальна`.
- Основной путь обновления должен идти через `updater.exe`: надстройка ставит обновление в очередь, а сам `updater.exe` после закрытия Excel скачивает и запускает нужный `setup.exe`.
- Блок выбора региона должен быть компактным: подпись сверху, dropdown под ней.
- Ширина dropdown региона должна ориентироваться на самый длинный вариант списка и не занимать лишнее место.
- Кнопки на вкладке должны быть разложены вручную по колонкам, а не отданы на произвольный перенос Excel.
- В каждом блоке должны использоваться подсказки `screentip/supertip`, чтобы длинные подписи не перегружали ленту.

## Рекомендуемая целевая архитектура

### 1. Одна рабочая надстройка и отдельный updater

Рекомендую две сущности:

- основная рабочая надстройка `MainVZID.xlam`;
- маленький `updater.exe`, который не участвует в каждой сессии Excel, а нужен только для обновления.

Бизнес-эффект:

- вкладка `VZID` живёт прямо в той надстройке, которую Excel реально загрузил как свою;
- исчезает лишний слой между Excel и рабочей надстройкой;
- запуск становится понятнее и стабильнее;
- обновление можно делать безопасно и контролируемо без самозамены файла "изнутри себя";
- при проблеме легче откатиться на прошлую версию;
- снижается риск антивирусных срабатываний по сравнению со схемой со скриптами, shell и постоянным загрузчиком.

### 2. Как должен вести себя правильный updater

Правильный updater не должен:

- писать и запускать `vbs/cmd/powershell`;
- тихо подменять файлы в `XLSTART`;
- тянуть и запускать внешние `.exe` без понятного пользовательского действия;
- вмешиваться в каждое открытие Excel;
- содержать бизнес-логику и ribbon.

Правильный updater должен:

- запускаться только по сценарию установки или обновления;
- понимать, где лежит установленный `MainVZID.xlam`;
- ждать полного завершения всех процессов Excel;
- проверять версию и контрольную сумму скачанного файла;
- делать резервную копию текущей рабочей версии;
- заменять файл только после полного закрытия Excel;
- уметь при неудаче оставить прошлую рабочую версию;
- писать простой лог результата;
- по выбору пользователя уметь снова открыть Excel после успешного обновления.

### 3. Как должно происходить обновление

Рекомендуемая схема:

- публикуется релиз в GitHub Releases этого же публичного репозитория;
- у релиза есть версия, файл надстройки и файл-манифест;
- `MainVZID` читает манифест;
- если версия новее локальной, показывает пользователю понятное действие:
  "Доступно обновление" / "Обновить сейчас" / "Напомнить позже";
- после подтверждения не трогает рабочий `MainVZID.xlam` напрямую, а ставит обновление в очередь;
- если текущего `updater.exe` достаточно, локальный `updater.exe` после закрытия Excel сам скачивает `setup.exe`, проверяет его целостность и запускает его;
- если релиз требует более новый `updater.exe`, надстройка использует переходный сценарий:
  сама скачивает `setup.exe`, проверяет его целостность и передает его локальному `updater.exe` как локальный файл для запуска;
- показывает понятное сообщение:
  "Обновление скачано. Оно установится только после полного закрытия всех окон Excel и нового запуска";
- предлагает:
  "Перезапустить Excel сейчас" / "Позже";
- `updater.exe` в фоне ждёт полного закрытия Excel, затем:
  либо делает резервную копию и заменяет `MainVZID.xlam`,
  либо запускает уже скачанный `setup.exe`,
  затем фиксирует результат в логе и при выборе пользователя запускает Excel заново;
- если GitHub недоступен, даёт пользователю резервный сценарий:
  установить обновление из локального файла релиза.

Дополнительно рекомендую:

- хранить как минимум одну прошлую рабочую версию локально;
- предусмотреть ручную кнопку "Проверить обновления";
- предусмотреть ручную кнопку "Установить обновление из файла";
- вести простой лог обновления для диагностики.
- показывать компактный индикатор состояния обновления на вкладке `VZID`, например:
  "Есть новая версия", "Обновление скачано", "Не удалось проверить обновления".
- в основной схеме автообновления `MainVZID` не скачивается и не подменяется напрямую из VBA;
- обновление рабочей надстройки и `updater.exe` должно идти через новый `setup.exe`.

### 4. Как должен выглядеть интерфейс

Рекомендую не "панель быстрого доступа", а отдельную вкладку ленты Excel.

Причина на уровне пользовательского поведения:

- вкладка ленты позволяет сделать нормальные группы кнопок, подписи, выпадающие списки и структуру по сценариям;
- панель быстрого доступа хороша для 5-10 избранных команд, но плохо подходит для большого продуктового интерфейса;
- в QAT неудобно строить понятную навигацию по регионам и наборам действий.

Простое объяснение:

- панель быстрого доступа — это маленькая полоска с несколькими иконками сверху;
- лента Excel — это большие вкладки вроде `Главная`, `Вставка`, `Данные`;
- для большого продукта почти всегда правильнее сделать свою вкладку на ленте, например `VZID`.

Предварительный целевой вид:

- вкладка `VZID`;
- выпадающий список выбора региона;
- группы кнопок по сценариям работы;
- отдельная группа "Сервис";
- отдельная группа "Обновление и диагностика".

Принцип кнопок для MVP:

- пользователь видит общий набор бизнес-кнопок;
- выбранный регион определяет, какие кнопки активны и какой макрос вызывается;
- если для региона сценарий не предусмотрен, кнопка остаётся видимой, но неактивной.

## Варианты интерфейса

### Вариант A. Основной интерфейс на вкладке ленты, сложные сценарии в отдельных окнах

Плюсы:

- быстро запускается;
- понятен пользователю;
- хорошо масштабируется;
- не требует перегружать экран сложными формами.

Минусы:

- для сложных сценариев всё равно остаются отдельные окна.

### Вариант B. Почти всё на вкладке ленты и боковых панелях/листах-настройках

Плюсы:

- меньше всплывающих окон;
- более современное ощущение работы.

Минусы:

- выше сложность реализации;
- в VBA это менее удобно и менее гибко, чем комбинация "лента + отдельные формы".

### Рекомендация

Начинать с варианта A.

## Первый согласованный MVP

### Структура вкладки `VZID`

- dropdown выбора региона;
- блок `Формирование сопроводительных документов`;
- блок `Действия`;
- блок `Обновление`.

### Визуальная раскладка вкладки

- блок выбора региона строится вертикально: текстовая подпись, затем dropdown;
- блок `Формирование сопроводительных документов` раскладывается в две колонки;
- в первой колонке должно быть до трех кнопок, во второй колонке оставшиеся кнопки блока;
- блок `Действия` для обычных регионов содержит одну колонку из трех кнопок;
- для региона `Тюмень` в блоке `Действия` должна появляться отдельная колонка из двух специальных кнопок;
- между колонками допускается только безопасная раскладка средствами RibbonX, без элементов, которые ломают загрузку вкладки;
- на все кнопки добавляются понятные пользовательские подсказки, которые позже можно редактировать без смены архитектуры.

### Блок `Формирование сопроводительных документов`

Этот блок должен быть одинаковым по составу кнопок для всех регионов:

- `Сформировать лист "Реестр пакетов в Excel"`
- `1 сопровод для WSS`
- `Итоговые сопроводы`
- `Заказной реестр`

Правило:

- если для региона у кнопки нет привязанного макроса, кнопка остается видимой, но неактивной.

### Блок `Действия`

Общие для регионов кнопки:

- `Отчет ручной обработки`
- `Подпись уведомлений/заявлений`
- `Извлечение постановлений об окончании БИ из PDF`

Дополнительно только для региона `Тюмень`:

- `Выбрать ОСП и Адрес`
- `Горячая клавиша`

Правило:

- тюменские специальные кнопки не должны показываться в других регионах вообще.

## Правило сохранения текущих окон

- `frmTerritory` как главный выбор территории убирается и заменяется на dropdown выбора региона на вкладке `VZID`.
- Логика отдельных региональных стартовых экранов (`frmVZID_KGN`, `frmVZID_TMN`, `frmVZID_EKB`, `frmVZID_CHLB`) переводится в команды и группы на вкладке `VZID`.
- Все прочие сценарии, где макрос уже сейчас открывает отдельный `UserForm`, по умолчанию сохраняются как есть.
- Пример сценария, который можно оставить без изменений по механике окна: `ManualProcReport`.

## Правило доступа к командам

- Если у пользователя нет доступа к команде, кнопка остаётся видимой.
- Такая кнопка отображается неактивной.
- Правила доступа определяются по Windows-логину.
- На старте эти правила хранятся в конфиге надстройки.

## Установщик первого запуска

Целевой сценарий:

1. Пользователь скачивает один `setup.exe`.
2. Запускает его без прав администратора.
3. Установщик сам раскладывает `MainVZID.xlam`, `updater.exe`, конфиг и служебные папки в профиль пользователя.
4. Установщик подключает `MainVZID.xlam` к Excel как пользовательскую надстройку.
5. После этого дальнейшие обновления идут уже штатно через систему обновления.

Рекомендуемые папки:

- `%AppData%\Microsoft\Excel\LocalCache\VZID\addin\`
- `%AppData%\Microsoft\Excel\LocalCache\VZID\updater\`
- `%AppData%\Microsoft\Excel\LocalCache\VZID\updates\`
- `%AppData%\Microsoft\Excel\LocalCache\VZID\backup\`
- `%AppData%\Microsoft\Excel\LocalCache\VZID\config\`
- `%AppData%\Microsoft\Excel\LocalCache\VZID\logs\`

## Что важно предусмотреть заранее

- где хранится выбранный регион;
- как разграничиваются команды по регионам;
- как пользователь понимает, какая версия сейчас установлена;
- как происходит первое подключение;
- как публикуется релиз в этом же публичном репозитории;
- нужна ли работа без интернета;
- есть ли пользователи за корпоративным прокси;
- нужна ли цифровая подпись VBA-проекта;
- нужен ли отдельный корпоративный сертификат;
- кто имеет право публиковать релизы.

## Нефункциональные требования

- минимизация срабатываний антивируса;
- отсутствие скрытых shell-скриптов и самозамены через системные команды;
- предсказуемый запуск;
- понятные сообщения об ошибках;
- логирование обновления и основных сбоев;
- безопасный откат на предыдущую версию.

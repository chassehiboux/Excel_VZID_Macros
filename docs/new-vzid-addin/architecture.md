# Архитектура: новая надстройка VZID

## Статус

Базовая архитектура после согласования продуктовых решений.

## Общая схема

Продукт состоит из трёх частей:

1. `Setup`
2. `MainVZID`
3. `Updater`

Все три части описываются и выпускаются из одного публичного репозитория:

- `https://github.com/chassehiboux/Excel_VZID_Macros`

## 1. Setup

### Назначение

Один `setup.exe`, который пользователь запускает без прав администратора.

### Что делает

- создаёт служебные папки в профиле пользователя;
- кладёт туда рабочую надстройку `MainVZID.xlam`;
- кладёт туда локальный `updater.exe`;
- кладёт стартовый конфиг;
- регистрирует `MainVZID.xlam` как обычную пользовательскую Excel-надстройку;
- не использует `Program Files`;
- не использует `XLSTART` как точку самоподмены.

### Целевые папки

- `%LocalAppData%\VZID\addin\`
- `%LocalAppData%\VZID\updater\`
- `%LocalAppData%\VZID\updates\`
- `%LocalAppData%\VZID\backup\`
- `%LocalAppData%\VZID\config\`
- `%LocalAppData%\VZID\logs\`

## 2. MainVZID

### Назначение

Основная надстройка со всей бизнес-логикой и пользовательской лентой.

### Что содержит

- вкладку `VZID`;
- dropdown выбора региона;
- кнопки бизнес-сценариев;
- привязку кнопок к макросам по региону;
- существующие отдельные `UserForm`, где они реально нужны;
- правила доступа к командам;
- сервисные команды;
- проверку обновлений и запуск `updater.exe`.

## 3. Updater

### Назначение

Маленький отдельный `updater.exe`, который не участвует в обычном запуске Excel и нужен только для безопасной замены файла надстройки после закрытия Excel.

### Что должен делать

- получать путь к скачанному обновлению и путь к установленному `MainVZID.xlam`;
- ждать полного завершения всех процессов Excel;
- проверить `sha256` и версию обновления;
- сделать резервную копию текущего рабочего файла;
- заменить рабочий `MainVZID.xlam`;
- зафиксировать результат в логе;
- при выборе пользователя повторно открыть Excel.

### Что не должен делать

- запускать `vbs`, `cmd`, `powershell`;
- жить как постоянный загрузчик при каждом старте Excel;
- выполнять бизнес-макросы;
- содержать ribbon и интерфейс продукта;
- скачивать произвольные файлы без сценария, инициированного `MainVZID`.

## Логика интерфейса

### Основные принципы

- у пользователя один вход через вкладку `VZID`;
- регион выбирается в dropdown;
- набор кнопок визуально стабилен;
- выбранный регион управляет доступностью и маршрутизацией команд;
- недоступные команды остаются видимыми, но неактивными.

### Общие кнопки MVP

- `Пакеты`
- `Сопроводительные`
- `Сопроводительные по получателю`
- `Заказные`

### Маршрутизация по регионам

#### Курган

- `Пакеты` -> `Doc_Packets`
- `Сопроводительные` -> `Make_Cover_Letters`
- `Сопроводительные по получателю` -> `Make_Cover_Letters_ByRecipient`

#### Тюмень

- `Пакеты` -> `Doc_Packets`
- `Сопроводительные` -> `TMN_Make_Cover_Letters_likeKGN`
- `Сопроводительные по получателю` -> `TMN_Make_Cover_Letters_ByRecipient_likeKGN`

#### Екатеринбург

- `Пакеты` -> `Doc_Packets`
- `Заказные` -> `RIC_Zakaznye_Create`

## Существующие формы

### Что убирается

- `frmTerritory` как главный экран выбора региона.

### Что заменяется вкладкой

- стартовая логика региональных экранов:
  `frmVZID_KGN`, `frmVZID_TMN`, `frmVZID_EKB`, `frmVZID_CHLB`.

### Что сохраняется

- прочие сценарии, где отдельное окно уже используется как рабочий инструмент;
- например `ManualProcReport`.

## Обновление

### Основной путь

Источник: GitHub Releases этого же публичного репозитория.

`MainVZID`:

- читает локальный метаданный файл;
- при необходимости запрашивает актуальный `manifest.json`;
- сравнивает версии;
- скачивает новый `.xlam`;
- проверяет `sha256`;
- запускает локальный `updater.exe`.

`Updater`:

- ждёт полного закрытия Excel;
- делает резервную копию текущего файла;
- заменяет `MainVZID.xlam`;
- пишет результат в лог;
- при необходимости заново запускает Excel.

В MVP автообновляется прежде всего `MainVZID`.

Сам `updater.exe` обновляется через новый `setup.exe`.

### Резервный путь

Пользователь вручную выбирает файл релиза через кнопку:

- `Установить обновление из файла`.

Дальше применяется тот же локальный `updater.exe`.

### Когда применяется обновление

- не в текущей живой сессии Excel;
- только после полного закрытия Excel;
- сразу после этого через `updater.exe`, а не в момент следующего старта внутри VBA.

### Что видит пользователь

- компактный alert на вкладке `VZID`;
- кнопку `Обновить`;
- кнопку `Установить обновление из файла`;
- сообщение после подготовки обновления:
  обновление будет установлено только после полного закрытия Excel.

## Конфиг

### Путь

- `%LocalAppData%\VZID\config\config.json`

### Что хранит

- выбранный регион;
- текущую активную версию `MainVZID`;
- путь к подготовленному обновлению;
- время последней проверки обновлений;
- статус последней проверки;
- правила доступа по Windows-логину;
- служебные флаги интерфейса.

## Права доступа

### Базовая модель

- проверка по Windows-логину;
- правила лежат в конфиге;
- `dzuikevich` имеет максимальные права по умолчанию.

### Поведение интерфейса

- если доступ есть, кнопка активна;
- если доступа нет, кнопка видна, но неактивна;
- при необходимости показывается поясняющая подсказка.

## Антивирусная стратегия

### Главные принципы

- никаких shell-скриптов для подмены файлов;
- никаких скрытых `vbs/cmd/powershell`;
- предсказуемая файловая структура;
- контролируемое скачивание файлов;
- проверка хэша скачанного релиза;
- отдельный маленький updater вместо VBA-самозамены файла;
- цифровая подпись должна быть предусмотрена как отдельный этап.

## Следующий уровень проектирования

После этого документа следующий шаг:

- расписать структуру проекта по папкам;
- определить модули `MainVZID`;
- определить контракт запуска `updater.exe`;
- определить формат `manifest.json`;
- определить состав `setup.exe`;
- зафиксировать реальный URL `manifest.json` в текущем публичном репозитории.
